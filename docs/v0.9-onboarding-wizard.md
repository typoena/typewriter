# v0.9 — On-device onboarding wizard (first boot + `:setup`)

> Part of the [Typoena macro plan](macroplan.md) — this doc expands the
> "on-device provisioning + settings screen" item of
> [v0.9-robustness.md](v0.9-robustness.md). Requirements and targets:
> [qfd.md](qfd.md). Load-bearing decisions: [adr.md](adr.md).

**Status:** slices 0–3 BUILT; slice 3 (device-flow sign-in) **verified
end-to-end on hardware 2026-07-16**. Slices 4–5 not started. Two Wi-Fi-step
refinements landed 2026-07-16 (host-tested, both firmware builds green): the
**password is shown by default** (Tab hides it) so a long random key isn't
entered blind, and the **SSID is picked from a device scan** rather than
typed. Token rename that landed alongside: the conf key is `TW_TOKEN` now
(legacy `TW_PAT` still parses). **Future direction (decided 2026-07-16):** a
**SoftAP companion webapp** will offer a phone-driven Wi-Fi hand-off (chosen
over BLE because Web Bluetooth is unsupported in iOS browsers) — this
reverses the no-SoftAP non-goal below; deferred, and the on-device keyboard
path stays as the no-phone fallback. The password still can't be
auto-shared (no browser API exposes a stored Wi-Fi password) — the webapp
only moves typing to the phone.

## Why

Today a Typoena can only be set up next to a Mac: the
[installer](../installer/DESIGN.md) provisions the SD card, and the firmware's
own config is **compiled in** from `firmware/.env` (ADR-007 — dev-only). The
wizard is the **zero-computer onboarding path**: a device fresh from the box,
powered on with a blank card, configures itself with only its own keyboard, its
panel, and the user's phone. It is a **peer** of the installer, not a
companion — two paths to the same configured card. It covers the no-Mac and
no-computer-at-all users, and it lets typoena.dev lead with "just power it on."

## Trigger

**Unconfigured = no `/sd/typoena.conf`** — not "card empty": fresh cards carry
junk (`System Volume Information`, `.Spotlight-V100`, …). The check lives in
`boot_storage` (`firmware/src/main.rs`), which already halts on no-SD and
no-repo; an unconfigured card branches into the wizard instead of
`boot_halt`. A configured card boots the editor exactly as today.

## Prerequisite — runtime config (slice 0)

The firmware does not read `typoena.conf` at runtime today: `git_sync.rs`
consumes seven `env!("TW_*")` compile-time consts baked by `build.rs` from
`firmware/.env`, and the conf the installer writes to the card is **read by
nothing**. Slice 0 fixes that inversion, and is what makes the *installer's*
card work for someone who is not the firmware author:

- A shared **`conf` crate** (host-buildable, no esp deps) owns the schema:
  the seven `TW_*` keys, required-ness, `KEY=value` parse/render, and
  `expand_remote_url` — ported from `installer/src/config.rs`, which the
  installer should later consume from the same crate (single source of truth;
  not done in slice 0 to keep the installer untouched).
- At boot the firmware parses `/sd/typoena.conf` into a `Conf` and threads it
  to the git thread; the `env!` consts become the **fallback** for values the
  card doesn't carry (dev builds keep working unchanged; `build.rs`'s
  four-var panic is relaxed to a warning since a git build no longer needs
  baked values to be useful).
- Precedence: **card overrides baked**. A field empty in both is caught by the
  existing runtime `bail!` guards in `publish_cycle`/`pull_cycle` (snackbar on
  first `:gp`/`:gl`), or by the wizard itself.

## The wizard

A step sequence rendered directly into a `display::Frame` (the
`show_message`/`boot_halt` pattern — no editor loop), consuming `keymap::Key`
from `usb_kbd::next_key()`. The step/field state machine lives in a
host-testable crate (`wizard/`, deps `keymap` + `display` + `conf`), with the
palette's `NewFile` input step as the editing model (Enter commits,
Backspace-past-start goes back). Every completed step persists atomically
(unlink + `*.tmp` + rename — FAT `f_rename` won't overwrite), so a power-pull
resumes at the right step, never in a half-configured boot.

1. **Welcome + Wi-Fi.** The device scans on entry and shows nearby networks
   as a **filterable pick list** (deduped, strongest-first) — the SSID is
   *selected*, not typed. Esc drops to manual entry for a hidden/missed
   network. Only the **password is typed**, on the device keyboard, and it is
   **shown by default** (Tab hides it) so a long random key isn't entered
   blind — justified by the device's own security model (secrets sit in
   cleartext on the card; physical custody is the control). Join is tested
   immediately (`net.rs::connect_wifi`, bounded retry) with a clear failure →
   re-edit loop. **Decision (revisited): no SoftAP/captive portal *in the
   on-device wizard*** — an embedded HTTP server + captive DNS + web UI is a
   whole firmware subsystem, and the scan-pick + reveal cover the keyboard
   step well. A SoftAP *companion webapp* is a separate, deferred path (see
   the status note) for users who'd rather not type the password on the
   device at all. Note: the modem currently moves into the git thread at
   boot; the wizard runs **before** that thread spawns, so it owns the modem
   during setup (scan + join) and releases it to the normal boot path when
   done. The radio is built once, started/stopped for the scan, then kept up
   from the join onward.
2. **Sign in with GitHub — the QR step.** The device runs the GitHub App
   device flow itself (client_id `Iv23liwgnE86ITDpBdnn`, same app as the
   installer's ^G): POST `login/device/code`, render the `verification_uri`
   as a **QR code on the e-paper** with the 8-char `user_code` printed under
   it, poll `login/oauth/access_token` at the server's interval, store the
   token. The phone is needed exactly once — scan, tap approve, done; the
   user never types a token. HTTPS via `EspHttpConnection` (the Spike 6
   client, promoted into the firmware proper with POST support);
   SNTP/trust-store setup reuses the `ensure_online` pieces. QR modules drawn
   as filled rects into the `Frame` (a QR generator is net-new — small
   pure-Rust encoder, no_std-friendly).
3. **Pick the repo.** GET the installation's repo list from the API, render
   as a keyboard-navigable list; identity (author name/email) prefilled from
   `/user`. Creating a repo is out of scope on-device — the GitHub App's
   installation page on the phone covers "grant a new repo" if needed.
4. **Clone — with a size gate.** The device clone is **shallow fetch
   (depth 1) + `apply_tree_diff(empty → tip)`** — `git init` at `/sd/repo`,
   set origin, fetch with `FetchOptions::depth(1)` (git2 0.20 / libgit2
   1.9.4), then the v0.7 O(changed) tree apply writes the working copy. The
   result is a **single pack** (the pack-not-loose state FAT wants) and media
   is skipped at checkout (`is_media_path`), exactly like `:gl`.
   **But libgit2 has no partial clone** (no blob filters), so tip media *is*
   downloaded even though it's never written. Hence the gate: before
   fetching, read `repo.size` (KB) from the API; **over ~30 MB the wizard
   refuses** with the honest message — *"this repo is too large to set up
   from the device — pick or create a smaller one, or seed the card from a
   computer once (typoena.dev)."* No workaround is pretended. (Future lever
   if ever needed: a pack-trimming proxy on typoena.dev infra — parked, real
   infra + token-trust cost.) Under the gate: progress on the panel
   (download, then per-file apply — ~1100 files is minutes on FAT-over-SPI,
   fine one-time with a progress line). Then seed `.typoena.toml`, the
   snippet catalog, and `/sd/local` if absent — the defaults move **into the
   firmware** (canonical copy; the installer should bundle from the same
   source so the two paths can't drift).
5. **Done.** Write the final conf, fall through to the normal boot path
   (note load, walk, git thread, editor).

## `:setup` — reset is the same wizard, prefilled

A second entry mode, not separate machinery: the `:setup` ex command (one arm
in `execute_command`, plus a `>` palette entry) reopens the wizard **prefilled
from the current conf** — the device-side derive ladder. Change Wi-Fi, re-auth,
or switch repos by skipping to the step you care about.

- **Config is rewritten only when a completed step changed something**
  (tmp + rename). Opening `:setup` and backing out is harmless — no
  delete-config-and-reboot.
- **Same repo re-chosen** → `/sd/repo` untouched, no re-clone, dirty journal
  survives.
- **Different repo, or factory wipe** → the **dirty guard is mandatory**: a
  non-empty `.typoena-dirty` journal or a stranded commit refuses with
  "publish first (`:gp`) or type the confirmation to discard" — installer
  parity (its dirty-card refusal), minus the backup offer (no second disk).
  The old tree is deleted **before** the new config is written, so a
  power-pull mid-delete lands back in the wizard, not in a boot pointing at a
  half-deleted repo. Tree delete over FAT is slow (~minutes at ~1100 files) —
  progress line required.
- **Factory wipe** (back to blank-card state) is a final confirmed option
  *inside* the wizard, not its own command.

## Token expiry (decision forced here)

A computer-less device with the GitHub App's default 8-hour expiring token
must store the refresh token and handle refresh — including
refresh-fails-while-offline, a whole failure mode. **Turn token expiry OFF in
the app settings** (the open item in the app setup) — with expiry off, the
device-flow token behaves like the fine-grained PAT it replaces.

## Slices

- [x] **0. Runtime conf** — `conf` crate (schema + parse/render +
      `expand_remote_url`, host-tested); firmware boot-reads
      `/sd/typoena.conf`, git thread consumes it, `env!` fallback retained,
      `build.rs` guard relaxed. Verifiable without any wizard UI: an
      installer-provisioned card drives `:gp`/`:gl` on a firmware built with
      a different (or empty) `.env`.
- [x] **1. Wizard core** — `wizard/` crate: step/field state machine, two
      entry modes (blank / prefilled), `Key`-driven editing, `Frame`
      rendering, resume-from-stage. Host-tested.
- [x] **2. Boot trigger + Wi-Fi step** — unconfigured branch in
      `boot_storage`; modem ownership during the wizard; join-test with
      re-edit on failure; atomic conf persistence per step.
- [x] **3. Device flow + QR** — BUILT 2026-07-15: `wizard::github` carries
      the pure protocol (form-encoded bodies/parsers ported from
      installer/src/auth.rs, `GET /user` JSON, host-tested); `wizard_io` does
      transport on `EspHttpConnection` + cert bundle after an SNTP gate, keeps
      Wi-Fi up for the whole wizard, and polls between keystrokes at GitHub's
      pace (slow_down honored, transport hiccups retry until the code's own
      deadline). The QR is real (`qrcodegen`, 4-module quiet zone, ~5 px
      modules). Failures park on an explicit retry screen (Enter retries,
      Backspace re-edits Wi-Fi) — no auto-retry radio/panel spin.
- [ ] **4. Repo pick + size gate + clone** — API list + `/user` identity;
      `repo.size` gate (~30 MB); init + shallow fetch + `apply_tree_diff`;
      progress; seed defaults (canonical in firmware).
- [ ] **5. `:setup`** — prefilled re-entry, write-on-change, dirty guard,
      delete-before-reconfigure, factory wipe.

## Non-goals

- SoftAP / captive portal / any embedded web UI.
- Creating GitHub repos from the device.
- Cloning large repos on-device (the gate refuses; the installer or a manual
  card seed covers them).
- Partial-clone proxy infrastructure (parked).
